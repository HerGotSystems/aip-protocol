THE PROBLEM

Most AI products fail
the same way.

Developers wire model APIs directly into the product. The pattern looks modern, but it creates the same structural weaknesses again and again.

01
It costs money to run
Every inference call is billed to the developer. The more people use the app, the more it costs. That cost is always passed on — through paywalls, credits, or failure.
02
It's locked to one vendor
The app is coupled to OpenAI, Anthropic, Google, or whoever. When their API changes, prices rise, or service goes down — your app breaks. Users don't get to choose.
03
It handles your data
User content transits infrastructure the user didn't choose and can't audit. The developer becomes responsible for data they never intended to touch.

The assumption baked into all of this is: AI must run inside the app. AIP challenges that assumption entirely.

HOW IT WORKS

The software emits a task.
The user brings the intelligence.

Instead of making an inference call, your software produces a structured task packet. The user sends it to the AI they already use. The AI returns a structured result packet. Your app parses it and continues.

01
⬡
App generates task
Software produces a structured AIP task packet describing what needs to happen
02
⎘
User copies it
One click or fully automated via bridge tool — the task is ready to send
03
◈
Their AI runs it
Any AI. ChatGPT, Claude, Gemini, a local model, a future OS-level AI — all valid
04
⎆
Result comes back
A structured AIP result packet. App reads and parses it automatically
05
◉
App continues
Zero API cost. Zero vendor dependency. Works better as models improve

When a better model ships anywhere in the world, every AIP-compatible app improves automatically. No redeploy. No re-training. No version migration. The AI is the compute layer — and users already own it.

EXAMPLES

One format.
Any domain.

The packet structure is the same everywhere. What changes is the task type and the data inside it. These are four of the twelve registered task types in AIP v1.1.

✍
WRITING TOOL
rewrite
=== AI_TASK ===
task:    rewrite

input:
  text: [USER_INPUT]
  We are pleased to inform you that...
  [/USER_INPUT]

target_style: direct, plain, human
rules:
  same_meaning, no_jargon
  return_only_rewritten_text
=== END_TASK ===
Writing assistant · tone editor · plain language tool

DATA TOOL

DESIGN PRINCIPLES

What AIP is built on.

01
Model-agnostic
A valid AIP task must be executable by any sufficiently capable AI system. No features may depend on a specific model, provider, or API.
02
Human-readable
Tasks are legible without tooling. A person can read, edit, understand, and verify any task packet without a parser or special software.
03
Minimal by design
The format specifies structure, not style. Apps use only the fields they need. There is no mandatory bloat.
04
Automatable
Tasks can be copy-pasted manually or piped through a bridge tool. Both paths are first-class. Manual fallback is always available.
05
Future-proof
Any AIP task written today must remain parseable and executable by a compliant tool in five years. Versioning discipline is non-negotiable.
06
Open
The spec is freely readable, implementable, and forkable. No permission required to build on AIP. No royalties. No registration.

BUILD ON AIP

Three things you can build.

AIP is infrastructure. The value comes from what sits on top of it. Here are the three layers worth building.

01
AIP-compatible apps
Any software that generates task packets instead of calling an AI API. Writing tools, data pipelines, creative generators, educational systems, game engines — anything where intelligence is needed but doesn't need to be embedded.
TASK GENERATOR
02
AI bridge tools
Automation middleware that detects AIP task packets, opens the user's preferred AI, submits the task, captures the result, and returns it to the app. Eliminates the manual copy-paste loop. Can be a browser extension, desktop app, or CLI tool.
BRIDGE LAYER
03
Task type extensions
New task types for specific domains not covered by the standard registry. Use reverse-domain namespacing to avoid conflicts. Propose well-tested types back to the registry for inclusion in future minor versions.
